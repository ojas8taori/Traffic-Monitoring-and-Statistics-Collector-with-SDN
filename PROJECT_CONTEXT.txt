KEY DECISIONS & DESIGN RATIONALE
================================================================================

1. TOPOLOGY CHOICE: Linear (h1 - s1 - s2 - h2, h3)
   Rationale: Minimal resource usage for WSL, sufficient for demonstrating
             controller-switch interaction and statistics collection.
   Alternative considered: Tree (more complex, higher resource usage)
   Decision: Linear chosen for 24-hour deadline

2. STATISTICS INTERVAL: 10 seconds
   Rationale: Balances visibility (frequent updates) with performance
   Alternative: 5s (too verbose) or 30s (too infrequent)
   Decision: 10s chosen

3. LEARNING SWITCH: Implemented in PacketIn handler
   Rationale: Standard SDN forwarding logic, demonstrates controller intelligence
   Alternative: Flood all traffic (simpler, less realistic)
   Decision: Learning switch chosen to show flow rules

4. FILTERING: Optional module (not core requirement)
   Rationale: Extends functionality for Scenario 2, shows advanced features
   Decision: Implemented as optional extension

5. LANGUAGE: Python 3 with Ryu framework
   Rationale: OpenFlow standard support, extensive documentation, WSL compatible
   Alternative: POX (alternative controller)
   Decision: Ryu chosen (compatibility issues being resolved)
